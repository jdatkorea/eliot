# [R&D-ADR] 2026-06-18: 브리핑 전달을 URL-hash 중복 페이로드에서 DB id-row 단일 저장으로 전환

## 1. Context & Motivation (배경과 문제 정의)
- **[Status]**: Accepted (구현·테스트·실서버 재현 완료)
- **[Trigger]**: 브리핑 A/B 듀얼 페이로드가 URL fragment(`#data=`)에 lz-string으로 압축되어 전달되고 있었는데, `urlA`·`urlB` 각각에 **동일한 dual payload가 중복 압축**되어 들어가는 구조였다. 실측 결과 이 중복이 Telegram `sendMessage` 4096자 한도를 가장 흔한 사용 경로(당일 5시간 트립, 4,326자)에서조차 넘기고 있었다 — 엣지 케이스가 아니라 기본 경로의 구조적 결함.

## 2. Architecture Decision (결정과 근거)
> **💡 재사용 추출 포인트 (Cross-Domain Pattern)**
> 상태를 클라이언트가 들고 다니는 URL에 직렬화하지 말고, 서버가 소유한 단일 진실 공급원(DB 1 row)을 만들어 클라이언트는 짧은 참조(id)만 들고 다니게 한다. payload 중복은 보통 "URL은 stateless해야 한다"는 압박에서 나온 임시방편인데, 데이터가 일정 규모를 넘으면 반드시 전송 계층의 숨은 한도(메시지 길이, 헤더 크기, URL 길이)에 부딪힌다 — 그 한도는 브라우저가 아니라 메시징 플랫폼(Telegram Bot API) 쪽일 수도 있다는 게 이번 사례의 교훈.

- **[Decision]**: `briefings` 테이블(`id text PK`, `payload jsonb`, `created_at`, `expires_at` 기본 7일 후) 신설. `buildBriefingLinks()`가 A+B dual payload를 **1행에 1회만** insert하고 짧은 id(base62 12자, `crypto.randomBytes` 기반, 신규 의존성 없음)를 반환하도록 변경. URL을 `/briefing/{id}?variant=A|B`로 축소(기존 1,859~3,645자 → 고정 62자). `app/briefing/page.tsx`(hash 파싱·lz-string 압축/해제·`useSyncExternalStore` 클라이언트 상태머신)를 전량 삭제하고, `app/briefing/[id]/page.tsx`(서버 컴포넌트, id로 1회 read 후 not-found 분기) + `BriefingView.tsx`(클라이언트, props로 전달받은 데이터만 렌더)로 계층을 분리.
- **[Rationale]**: 부분 완화안(A/B가 각자 자기 payload만 갖도록 중복만 제거)도 검토했으나, 그래도 fragile한 hash-decode 경로 자체는 남기 때문에 기각. AI/외부 API를 끌어들이는 대안은 검토 대상도 아니었음 — 순수 구조(전달 메커니즘) 변경만으로 해결되는 문제였음. 서버 컴포넌트로 read를 강제함으로써 "런타임 DB read ≤2" 불변식을 코드 구조 자체로 보장했고, 클라이언트의 추가 비동기 fetch hop을 없앴다(이전엔 클라이언트가 hash 파싱 후 동기적으로 디코딩 — 별도 round trip은 없었지만 fragile한 직렬화 경로였음).

Telegram WebApp CloudStorage(클라이언트 로컬 SDK) 호출이라 외부 네트워크 API는 아니지만, `lib/engine/` 폴더 경계 위반은 사실. 이번 세션에서 고치지 않음(다음 세션 우선 후보, §4).

- **[Rule 3: Data Integrity]**: `places` **537행** (실측, anon REST 직접 조회). 컬럼: `id, destination, name, category, is_outdoor, no_kids_zone, tags, stroller_friendly, has_nursing_room, last_social_seen` — `lat`/`lng` 없음(과거 DROP 확정 유지). 중복 0건(이번 세션 T0에서 `(destination,name)` 키 기준 21건 dedup + 인덕원_권역 279행 purge + 비-장소 노이즈 13건 제거, 850→537행). `briefings`(신규) **0행**(검증용 테스트 행은 세션 종료 전 정리 완료, RLS anon select/insert 정상 동작 확인). `app_config` 4키 시딩됨. `destinations`(DB 테이블) 0행 — 여전히 미사용, 좌표 매칭은 빌드타임 정적 JSON(`data/destinations.json`, `data/region-tiers.json`)으로 대체 동작 중.
- **[Test 실행 결과]**: `npx vitest run` → **19 suites / 204 tests 전부 통과**. `tsc --noEmit` → 이번 세션 변경분 기준 신규 에러 0건(이번 세션이 손대지 않은 파일 2곳의 기존 무관 에러만 잔존).

# [R&D-ADR] 2026-06-18: region-tier 게이트·날씨 하드제외·clock-time 축·조건부 체크리스트를 동일한 production-rule 패턴으로 통일하고 day-trip 엔진을 freeze

## 1. Context & Motivation (배경과 문제 정의)
- **[Status]**: Accepted (T2~T5 전 구간 구현·테스트·실측 완료, freeze 확정)
- **[Trigger]**: 직전 ADR의 `[Next Action]`이 그대로 트리거였다 — `passesRegionGate()`가 `extend_range` 태그 시 전체 풀로 무조건 우회해 B·확장형 링크가 영구히 전국(EXCLUDED 포함) 풀에 노출되는 구조적 결함. 이를 닫는 과정에서 같은 클래스의 잠재 결함 3개가 연쇄로 드러났다: 날씨 신호가 위험을 오히려 은폐하는 inversion(SPOF3, 직전 ADR §6에서 이미 실측 확정), 코스에 시각 축이 없어 야외 일정이 일몰을 넘길 수 있는 구조(T4 신설 범위), 체크리스트·이동수단 안내가 `extend_range`(T2 이후 의미가 바뀐 플래그)에 그대로 키잉되어 있던 잠재 버그(T5 작업 중 발견).

## 2. Architecture Decision (결정과 근거)
> **💡 재사용 추출 포인트 (Cross-Domain Pattern)**
> (1) 도메인이 다른 네 가지 문제(지역 게이트·날씨 제외·시간 배분·체크리스트 분기)를 전부 같은 `{when, then}` config-DSL 모양으로 풀었다 — 점수 가중치나 ad-hoc if문 대신 "조건 충족 시 배제/포함"이라는 동일 형태를 반복 사용하면, 새 규칙 추가가 코드 변경이 아니라 데이터(config row) 추가가 된다. (2) 더 일반적인 교훈: **플래그의 의미는 그 플래그를 만든 시점에 고정되지 않는다.** `extend_range`는 원래 "전국 우회"를 의미했으나 T2가 CAPITAL_EXT cap을 걸면서 "근거리 확장"으로 의미가 바뀌었고, 그 사실을 모르는 다른 소비처(`resolveTransportAdvice`, 체크리스트)는 옛 의미로 계속 키잉되어 있었다 — 플래그 의미를 바꾸는 변경은 "이 플래그를 읽는 모든 곳"의 명시적 재감사를 강제해야 한다(타입체커는 이런 의미론적 드리프트를 잡지 못한다).

- **[Decision]**:
  - **T2**: `passesRegionGate()`를 tier 기반(ICN_METRO/CAPITAL_EXT/EXCLUDED)으로 재작성. `buildActiveDestinationPlan()` — 단일 destination 우선 + 최근접-centroid spillover(최대 2개, `MAX_SPILLOVER_DISTANCE_KM=40km` 캡, two-pass: 반경 내 우선 탐색 후 tier 전체로 fallback)로 0-stop을 방지하면서 EXCLUDED 누출을 차단.
  - **T3**: `parseWeatherFromText()`의 `rain_prob` 강제 하향(inversion) 제거, `heatwave`/`coldwave`/`uv_high`를 독립 `conditions` 배열로 분리. `weather_exclusion_rules`(`{when:{weather_condition,is_outdoor}, then:{exclude:true}}`) config DSL을 `course-generator.ts`/`swap-spot.ts` 양쪽의 풀 필터링에 하드 적용(단순 안내 문구가 아니라 실제 배제), live `app_config`에 시딩.
  - **T4**: `phase_durations`(상대 가중치, 절대 분 아님)를 작전시간에 비례 배분하는 `resolvePhaseClockWindows()` 신설. 일몰을 넘기는 야외 phase는 *사후에 swap*하지 않고 *선택 시점에* `indoorOnly`를 강제(기존 `indoor_only` mood tag와 동일한 메커니즘 재사용).
  - **T5**: 체크리스트의 "여권·항공" 항목을 `extend_range` 대신 region-tier 기반 `isDayTripNear(destination, trip_days)`로 재키잉(당일·ICN_METRO/CAPITAL_EXT만 하드 제외). `stroller_friendly`(family 모드 `weightedScore` 가산점) · `has_nursing_room`(`Block.care_note`="수유실 완비" 노출) — 둘 다 ingest 시점부터 DB에 존재했지만 엔진 전체에서 소비가 0건이었음을 grep으로 먼저 확인한 뒤 추가.
  - **INVARIANT 정리**: `resolvePriorFeedback()`(`await getFeedback()`)을 `lib/engine/trip-context.ts`에서 `lib/webapp/feedback-storage.ts`로 이동 — 유일한 호출부(`submit-trip-request.ts`)가 이미 엔진 호출 *이전에* await해서 순수 데이터만 넘기고 있었으므로 동작 변경 없는 파일 위치 수정. `__tests__/engine-purity.test.ts`(Node fs 직접 스캔, 외부 `rg` 비의존)로 "`lib/engine/`에 `fetch(`/`await ` 0건"을 영구 회귀 가드로 고정.

- **[Rationale]**: 네 문제를 각각 다른 패턴(스코어링, 텍스트 안내, 하드코딩 분기 등)으로 풀 수도 있었으나, 기각했다 — 이미 확립된 production-rule 패턴(`{when,then}`)을 재사용하면 검증 방법(grep으로 규칙 존재 확인, 단위 테스트로 배제 동작 확인)도 재사용되고, 다음 사람이 새 규칙을 추가할 때 엔진 코드를 읽지 않고 config 모양만 보고 패턴을 추론할 수 있다. `isDayTripNear`를 mood_tag가 아니라 region tier로 재정의한 것은 "체크리스트가 실제로 보장하려는 안전 속성(원거리·숙박 시 여권 필요)"에 직접 대응하는 신호를 쓴 것 — `extend_range`는 그 속성의 대리 신호였을 뿐이고, T2가 그 대리 신호의 의미를 바꿔버린 순간 더 이상 신뢰할 수 없었다.
