# [R&D-ADR] 2026-06-18: Zero-Noise UX — Telegram 단일 진입 링크·브리핑 뷰 아코디언 스왑으로 전달·조작 계층 분리

## 1. Context & Motivation (배경과 문제 정의)
- **[Status]**: Accepted (구현·테스트·프로덕션 배포 완료)
- **[Trigger]**: 직전 ADR(part B)의 `[Next Action]`은 ingest-time Validator였으나, day-trip 엔진 freeze 직후 운영 경로에서 **전달 계층의 인지 부하**가 더 즉각적인 UX 결함으로 부상했다. part B에서 DB id-row 전환으로 Telegram 메시지 길이는 4,326→724자로 줄었지만, 봇은 여전히 A/B **두 개의 인라인 링크**를 보내고 웹뷰는 상단 **A/B 탭 + 항상 노출된 스왑 버튼**으로 시각적 노이즈가 남아 있었다. 엔진·데이터 불변식은 동결된 상태에서, 백엔드 로직을 건드리지 않고 **순수 UI·메시지 조립 레이어만** 압축하는 것이 이번 세션의 결정적 원인이었다.

## 2. Architecture Decision (결정과 근거)
> **💡 재사용 추출 포인트 (Cross-Domain Pattern)**
> 듀얼 옵션(변형)을 지원할 때, **선택 UI는 이미 전체 페이로드를 소유한 계층(웹뷰)**에만 두고, 상위 채널(메시징·푸시)은 단일 진입점 + 짧은 안내 문구만 남긴다. 채널마다 동일한 분기(링크 2개, 탭 2개)를 복제하면 각 채널의 한도·레이아웃 제약에 따로 맞춰야 하고, 사용자는 "어디서 비교해야 하는지"를 매번 재학습한다.

- **[Decision]**:
  - **Telegram(webhook 레이어)**: `buildTelegramLinkMessage()`가 `urlA` 단일 `<a>` 링크 + `COURSE_COMPARE_HINT`("💡 웹뷰에서 두 가지 코스 옵션을 비교할 수 있습니다.")만 전송. `urlB`·`labelA`·`labelB` 기반 A/B 분리 링크 문구 제거. `buildBriefingLinks()`·`relay-briefing.ts`는 여전히 A/B URL을 생성하지만(웹뷰 variant 쿼리·내부 테스트용), **사용자 대면 메시지에서는 A안 URL만 노출**.
  - **브리핑 웹뷰(UI 레이어)**: `BriefingView.tsx` — A/B **탭 → 컴팩트 드롭다운**(`variantMenuOpen`, `history.replaceState`로 `?variant=` 동기화). 스왑 버튼 **기본 숨김** → 블록 카드 탭 시 `expandedBlockKey` 아코디언 확장 후에만 "다른 곳으로" 노출. 공통 `components/ui/Button.tsx`·`Card.tsx`·`cn.ts` 추출, `WebAppForm.tsx`도 동일 컴포넌트로 정리.
  - **백엔드·엔진**: `/api/course/swap`, `buildBriefingLinks()`, DB id-row read 경로 **무변경** — 순수 프레젠테이션 계층 수정만.

- **[Rationale]**: Telegram에서 B 링크를 제거해도 dual payload는 DB 1행에 그대로 있고, 웹뷰 드롭다운이 variant 전환을 담당하므로 기능 회귀 없음. 메시지에 링크 2개를 유지하는 대안은 길이·스캔 비용만 줄었을 뿐 "비교는 웹뷰에서"라는 단일 멘탈 모델을 깨뜨려 기각. 스왑을 항상 노출하는 대안은 모바일 세로 화면에서 블록당 버튼 4~8개가 상시 점유해 기각 — 아코디언은 **의도적 조작(progressive disclosure)** 패턴으로 동일 API 호출을 유지한다.
