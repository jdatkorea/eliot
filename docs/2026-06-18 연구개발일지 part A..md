## 5. [실측] 도출된 문제점·SPOF (정확 기전 포함)

### SPOF1 — briefing 전달 경로: URL-hash 페이로드 잔존
`lib/webhook/briefing-urls.ts:134-136`의 `buildBriefingUrl()`은 지금도 `/briefing#data=<lz-string압축JSON>&variant=`를 생성하고, `app/briefing/page.tsx`는 지금도 `window.location.hash`를 파싱한다(`parseHashParams`/`decodeBriefingPayload`). `07c5b22`("A2/A3 refactor and RPC migration")는 `feedback_events`/`app_config` 조회만 RPC로 옮긴 것으로, **브리핑 본문 전달 경로 자체는 미수정**.
**[미검증]** 실데이터로 직접 생성해 길이 측정·렌더 실패 재현을 하지 않았다 — 길이초과/렌더실패의 실제 원인은 **미확정**(이하 "img5 원인 미확정"으로 표기). 수정 전 재현 선행 필요.

### SPOF2 — 지리 오염: "region 필터 부재"가 아니라 "확장형=전국풀" 의미론
**[정정 — 변경 전→후]**
- **변경 전(기존 계획 전제)**: destination 기반 region 필터가 존재하지 않거나 약해서 원거리 데이터가 혼입된다고 가정.
- **변경 후(실측 확정)**: `passesRegionGate()`(`course-generator.ts:146-153`)는 **존재**하며 정상적으로 정확 일치 검사를 수행한다. 문제는 `deriveVariantB()`(`variant.ts`)가 **항상** B-변형에 `extend_range` 태그를 추가하고, `passesRegionGate()`는 `extend_range` 포함 시 **무조건 true**를 반환해 게이트를 완전히 우회한다는 점이다(`course-generator.ts:150-151`). 결과적으로 모든 사용자가 받는 두 링크 중 "B·원거리·확장형" 링크는 가족의 실제 위치와 무관하게 **항상 850행 전체 풀**(경주·부산·제주 포함)에 노출된다. 이는 버그가 아니라 "B=확장형"이라는 기존 설계 의도지만, 그 설계가 곧 "원거리 혼입" 증상의 정확한 발생 기전이다.
- fix 방향: `extend_range`를 무조건 전체 우회가 아니라 **CAPITAL_EXT까지로 cap**하고, variant A(근거리)가 ICN_METRO만 쓰는지 별도 정합성 확인 필요.

추가로 destination 문자열 정규화 결함이 동일 SPOF의 일부로 발견됨: `canonicalizeDestination()`(`course-generator.ts:136`)은 `_근교` 접미사만 정규화하고 **`_권역` 접미사는 처리하지 않음** — `인덕원_권역`(279행, 전체의 32.8%)이 canonical `인덕원`과 영구히 정확 매칭 불가능한 상태로 방치됨(§5 데이터 부패 참조).

### SPOF3 — 날씨 규칙: 단순 부재가 아니라 역방향(inversion) 버그
**[정정 — 변경 전→후]**
- **변경 전(기존 계획 전제)**: 폭염 등 기상 조건에 대한 hard exclude 규칙이 단순히 구현되지 않았다고 가정.
- **변경 후(실측 확정)**: `parseWeatherFromText()`(`lib/engine/trip-context.ts:27-42`)는 `폭염|한파|자외선` 키워드를 정규식으로 감지하지만(line 31), 감지되면 `rain_prob`를 **"10%"로 낮춘다**(line 39, `rain_prob: rainLike ? "70%" : heatLike ? "10%" : "30%"`). 유일한 날씨 경고 트리거는 `rain_prob ≥ rain_prob_threshold`(기본 50%) 조건뿐이므로(`generate-briefing.ts:139`, `apply-course-swap.ts:59-63`), **폭염 신호가 경고 발생 가능성을 오히려 낮추는 역설**이 코드에 존재한다. `is_outdoor ∩ 폭염` 하드 제외 규칙은 코드 전체에 **없음**(grep으로 확정).
- fix 방향: line 39의 inversion 제거 + `is_outdoor ∩ 폭염` hard exclude를 config DSL로 추가.

### 데이터 부패 (SPOF1~3과 별개, 독립 발견)

**[정정 — slug/PK 관련, 변경 전→후]**
- **변경 전(기존 계획 전제)**: slug를 PK에서 격하하고 DB가 id를 소유하도록 "추가 작업"이 필요하다고 가정, slug CHECK constraint 도입 예정.
- **변경 후(실측 확정)**: **이미 완료된 상태**다. DB의 실제 PK는 CSV 단계의 `destination+slug` 합성키를 `uuidv5`로 해시한 UUID이며(`scripts/lib/place-sync.ts:toPlaceUuid`), slug는 처음부터 CSV 전용 컬럼으로 DB 스키마에 존재하지 않는다. 따라서 **slug CHECK constraint 항목은 폐기**한다. T0는 "PK 격하"가 아니라 "dedup + 인덕원_권역 purge + CSV 합성키 안정화(slug 포맷 혼재로 인한 향후 합성키 충돌/유실 방지)"로 재정의한다.
