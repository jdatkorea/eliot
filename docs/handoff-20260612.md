# ELIOT 프로젝트 최종 이관 문서

**작성일:** 2026-06-12  
**상태:** P0 ~ P2 M3(Config Decoupling) 완료 · 시스템 운영 안정화 단계 진입  
**대상 독자:** 다음 개발 세션 담당자, 운영자(Commander)

---

## 핵심 성과 요약

ELIOT은 **코드 하드코딩 기반 운영**에서 **데이터(Google Sheets) 기반 운영**으로 전환되었다.

| 이전 | 현재 |
|------|------|
| 장소·설정을 TypeScript 파일에 직접 수정 후 배포 | Google Sheets에서 SSOT(Single Source of Truth) 관리 |
| 설정 변경마다 코드 PR·배포 필요 | `cms:sync`로 Supabase 스냅샷 갱신 후 즉시 반영 |
| Engine이 하드코딩 config에 결합 | Engine은 순수 함수 유지, `AppConfig`를 외부 주입 |

운영자(Commander)는 코드를 건드리지 않고 Sheets만 수정한 뒤 동기화 스크립트를 실행하면 된다. 런타임(TRIP-PREP 웹훅)은 Supabase 스냅샷만 읽으며, Google API 호출은 SEED 단계에서만 수행된다.

## 1. 시스템 아키텍처 요약

### 1.1 3-Phase 파이프라인

| Phase | 실행 시점 | 역할 | 외부 API |
|-------|-----------|------|----------|
| **SEED** | PC, 목적지당 1회(또는 데이터 변경 시) | Sheets/fixture → Supabase upsert | Google Sheets API (SEED only) |
| **TRIP-PREP** | Telegram 웹훅, 여정 당일 | Supabase read → Engine×2(A/B) → 정적 브리핑 링크 전송 | LLM/유료 API 0건 |
| **FEEDBACK** | 여정 종료 후 | 5척도 + `failure_reason` → `feedback_events` 1 insert | — |

### 1.2 아키텍처 불변식 (Immutable Rules)

| # | 규칙 | 구현 근거 |
|---|------|-----------|
| 1 | **런타임 Google API 호출 0건** | `googleapis`는 `scripts/sync-sheets.ts`에만 존재 |
| 2 | **Sheets → DB 단방향** | Sheets = SSOT, Supabase = 배포 스냅샷 |
| 3 | **Engine 순수 함수** | `generateBriefing()` — IO 없음, `config`·`places`·`feedback_events`를 인자로 주입 |
| 4 | **웹훅은 Supabase만 조회** | `fetchBriefingData()` 경로 고정, 3초 timeout 후 fixture fallback |

**추가 PRD 불변식:**

- TRIP-PREP 단계 LLM/유료 API 0건
- `callback_query` 기반 상태관리 금지
- clock time 표기 금지 (`TimeLabel` enum만 사용)
- Safe Pool 밖 장소 발명 금지
- 무상태 확정: A/B 둘 다 렌더·전송, "확정"은 운영자가 링크 선택하는 행위

### 1.3 순수 함수 구조

```
TripRequest
  → normalize()                         [lib/engine/normalize.ts]
  → generateBriefing({                   [lib/engine/generate-briefing.ts]
       normalized, places, feedback_events, config
     })
  → deriveVariantB() / variantLabel()   [lib/engine/variant.ts]
  → apply-config 헬퍼                   [lib/engine/apply-config.ts]
       resolveMoodEffects, weatherKeyFromRainProb, fillDescTemplate
```

P2 M3 이후 `generateBriefing`은 `config: AppConfig` **필수 매개변수**를 받는다. 함수 내부에서 비동기 호출·외부 Fetch를 수행하지 않는다.

### 1.4 주요 모듈 맵

| 경로 | 역할 |
|------|------|
| `app/api/webhook/telegram/route.ts` | TRIP-PREP 웹훅 진입점 |
| `app/webapp/` | Telegram Mini App 입력 UI |
| `app/briefing/page.tsx` | hash 기반 정적 브리핑 렌더러 |
| `app/feedback/` + `app/api/feedback/` | 피드백 폼·Write API |
| `lib/engine/` | 코어 순수 함수 (generate, normalize, variant, apply-config, types) |
| `lib/config/app-config.ts` | `AppConfig` 타입, Zod, `DEFAULT_APP_CONFIG`, fail-over |
| `lib/config/mood-tags.config.ts` | WebApp용 mood tag re-export (DEFAULT 기반) |
| `lib/supabase/fetch-briefing-data.ts` | 런타임 I/O — places + feedback + app_config `Promise.all` 병렬 fetch |
| `lib/webhook/` | URL 생성, Telegram 파싱·메시지 포맷 |
| `lib/feedback/` | 동적 피드백 URL·`context_tags` 변환 |
| `lib/seed/validate-places.ts` | Sheets → Place Zod 검증 |
| `scripts/` | SEED 전용 (sync, seed, dry-run, test-webhook) |
| `fixtures/places.sample.json` | v0 Safe Pool 10곳 샘플 |

**기술 스택:** Next.js 16, React 19, Supabase, Vitest, Zod, googleapis (devDependency, SEED only)

### 1.5 격리 검증 체크리스트

- [x] `app/`, `lib/engine/`, `lib/webhook/`에 `googleapis` import 없음
- [x] 웹훅 → `fetchBriefingData()` only
- [x] Engine에 DB/CMS 직접 의존 없음 (config는 인자 주입)
- [x] `cms:sync`는 `scripts/` 트리에만 존재

### P2 M3 — Config Decoupling ✅ (최종 마일스톤)

**목표:** 하드코딩 `templates.config.ts` 제거 → Sheets `config` 탭 → Supabase `app_config` → Engine 주입

| 산출물 | 경로 | 설명 |
|--------|------|------|
| DB 마이그레이션 | `supabase/migrations/20260612_app_config.sql` | key-value + JSONB |
| Config 모듈 | `lib/config/app-config.ts` | `AppConfigSchema`, `DEFAULT_APP_CONFIG`, `safeAppConfigFromDbRows` fail-over |
| Config 동기화 | `scripts/lib/config-sync.ts` | `parseConfigFromSheet`, `upsertAppConfig` |
| 통합 sync | `scripts/sync-sheets.ts` | places + config 병렬 fetch/parse/upsert |
| I/O 레이어 | `lib/supabase/fetch-briefing-data.ts` | 3테이블 `Promise.all` 병렬, 3s timeout |
| Engine 리팩토링 | `lib/engine/generate-briefing.ts` | `config` 파라미터 의존, `apply-config.ts` 분리 |

| Dry-run | `scripts/dry-run-config-validation.ts` | Google API 없이 config 무결성 시뮬레이션 |

**필수 config keys:** `mood_tags`, `mood_tag_effects`, `templates`, `rain_prob_threshold`

**P2 M3 절대 원칙 (준수 완료):**

1. `generateBriefing` 내부 비동기/외부 Fetch 금지 — config는 매개변수 주입
2. Supabase read 시 `Promise.all` 병렬화

**품질 현황 (2026-06-12):** Vitest **36건 전체 통과** (6 test files)

## 3. 현재 시스템 데이터 흐름

### 3.1 전체 흐름도

```
┌─────────────────────┐     SEED only      ┌──────────────────────────┐
│  Google Sheets      │ ─────────────────► │  scripts/sync-sheets.ts  │
│  tabs: places,      │   googleapis       │  + lib/place-sync.ts     │
│        config       │                    │  + lib/config-sync.ts    │
└─────────────────────┘                    └──────────┬───────────────┘
                                                      │ upsert (SYNC_EXECUTE=true)
                                                      ▼
                                           ┌──────────────────────────┐
                                           │  Supabase                │
                                           │  • places                │
                                           │  • app_config            │
                                           │  • feedback_events       │
                                           └──────────┬───────────────┘
                                                      │ fetchBriefingData()
                                                      │ Promise.all, 3s timeout
                                                      ▼
┌─────────────────────┐                    ┌──────────────────────────┐
│  Telegram Webhook   │ ◄── buildBriefingLinks ──│  generateBriefing (pure) │
│  /briefing#data=…   │                    │  config 주입              │
└─────────────────────┘                    └──────────────────────────┘
```

### 3.2 places 탭 (Safe Pool)

1. `fetchSheetValues()` — `GOOGLE_SHEETS_PLACES_RANGE` (기본 `places!A1:O`)
2. `parsePlacesFromSheet()` — 헤더 동적 매핑 + `SheetPlaceSchema.safeParse()`
3. `upsertPlaces()` — UUID v5 결정적 ID, `onConflict: id`

**제외 조건:** `id` 빈 값, `status=archived`, Zod 검증 실패 행

### 3.3 config 탭 (운영 설정, P2 M3)

| 컬럼 | 설명 | 예시 |
|------|------|------|
| `key` | 설정 키 | `mood_tags` |
| `value` | JSON 또는 스칼라 | `["relaxed","food_light",...]` |
| `scope` | 적용 범위 | `global` |
| `updated_at` | 수정일 | `2026-06-12` |

**흐름:**

1. `GOOGLE_SHEETS_CONFIG_RANGE` (기본 `config!A1:D`)
2. `parseConfigFromSheet()` — `scope=global`만, JSON 구문 오류 행 스킵
3. `upsertAppConfig()` — `onConflict: key`
4. 런타임: `safeAppConfigFromDbRows()` — 필수 키 누락/Zod 실패 시 `DEFAULT_APP_CONFIG` fail-over

### 3.4 대체 SEED 경로 (fixture)

Google API 없이 로컬 개발·CI에서 사용:

```bash
pnpm run db:seed
```

`fixtures/places.sample.json` → 동일 `upsertPlaces()` 경로. config는 `DEFAULT_APP_CONFIG` 사용.

### 3.5 런타임 fallback

Supabase 미설정·3초 timeout·오류 시 → `getFixtureBriefingData()` (`lib/fixtures/briefing-data.ts`)

### 3.6 TRIP-PREP 상세 시퀀스

```
Telegram WebApp POST
  → parseWebhookBody()
  → fetchBriefingData()          # places + feedback_events + app_config
  → buildBriefingLinks()         # generateBriefing × 2 (A/B)
  → buildFeedbackUrl()           # trip_id = randomUUID()
  → sendTelegramLinks()          # TELEGRAM_BOT_TOKEN 필요
  → Response.json({ urls, trip_id, feedback_url })
```

## 4. 운영자(Commander) SOP

### 4.1 일상 운영 절차

#### Step 1 — Sheets에서 데이터 수정

- **장소 추가/수정:** `places` 탭
- **운영 설정 변경:** `config` 탭 (mood tags, templates, origin 좌표 등)
- 서비스 계정 이메일에 스프레드시트 **뷰어 이상** 권한이 있어야 함

#### Step 2 — Dry-run 검증 (권장)

```bash
pnpm run cms:sync:dry-run          # places Zod 검증 (Google API 불필요)
pnpm run cms:sync:config-dry-run   # config JSON 무결성 검증 (Google API 불필요)
```

#### Step 3 — 실제 동기화

```bash
# PowerShell
$env:SYNC_EXECUTE="true"; pnpm run cms:sync

# Bash
SYNC_EXECUTE=true pnpm run cms:sync
```

기본값 `SYNC_EXECUTE=false` — 읽기·파싱만 수행, upsert 없음.

#### Step 4 — 동작 확인

```bash
pnpm run test:webhook    # 웹훅 E2E 콘솔 출력
pnpm test                # 유닛 테스트 36건
```

### 4.2 npm 스크립트 레퍼런스

| 명령 | 용도 |
|------|------|
| `pnpm dev` | 로컬 Next.js 개발 서버 |
| `pnpm test` | Vitest 전체 (36건) |
| `pnpm run db:seed` | fixture → Supabase places upsert |
| `pnpm run cms:sync` | Sheets → Supabase (기본 dry-run) |
| `pnpm run cms:sync:dry-run` | places Zod 부분동기화 검증 |
| `pnpm run cms:sync:config-dry-run` | config JSON 무결성 검증 |
| `pnpm run test:webhook` | 웹훅 E2E 콘솔 출력 |

### 4.3 환경 변수 (`.env.local`)

```env
# Supabase (런타임 + SEED)
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=          # SEED 스크립트 전용

# Google Sheets (SEED only)
GOOGLE_SERVICE_ACCOUNT_KEY={"type":"service_account",...}  # 한 줄 JSON
GOOGLE_SHEETS_SPREADSHEET_ID=
GOOGLE_SHEETS_PLACES_RANGE=places!A1:O      # 선택
GOOGLE_SHEETS_CONFIG_RANGE=config!A1:D     # 선택

# 동기화 실행 게이트
SYNC_EXECUTE=true                   # false(기본) = 읽기·파싱만

# Telegram / 앱
TELEGRAM_BOT_TOKEN=
TELEGRAM_CHAT_ID=                   # 선택
APP_BASE_URL=http://localhost:3000
```

### 4.4 장애 대응 매트릭스

| 증상 | 원인 | 대응 |
|------|------|------|
| `cms:sync` 후 DB 변경 없음 | `SYNC_EXECUTE` 미설정 | `SYNC_EXECUTE=true`로 재실행 |
| `[skip] 행 N / …` 다수 | Zod 검증 실패 | 로그 사유 확인 → Sheets 셀 형식 수정 (`docs/cms-architecture.md` 변환 규칙 참조) |
| `[config JSON 오류]` | config value JSON 깨짐 | 해당 행만 스킵됨 — JSON 수정 후 재동기화 |
| `[config 경고] 필수 config key 누락` | 필수 5키 미완 | upsert는 진행되나 런타임은 `DEFAULT_APP_CONFIG` fail-over |
| `GOOGLE_SERVICE_ACCOUNT_KEY가 … 없습니다` | env 누락/JSON 파싱 실패 | `.env.local` 한 줄 JSON 확인 |
| `NEXT_PUBLIC_SUPABASE_URL 또는 SUPABASE_SERVICE_ROLE_KEY가 …` | SEED env 누락 | service role key 설정 |
| 웹훅이 fixture 데이터 사용 | Supabase 미설정 또는 3s timeout | env 확인, Supabase 연결·테이블 존재 확인 |
| Telegram 메시지 미발송 | `TELEGRAM_BOT_TOKEN` 없음 | env 설정 (웹훅 JSON 응답은 여전히 반환) |
| places ID 불일치 | 슬러그→UUID v5 매핑 | 동일 슬러그는 항상 동일 UUID (`PLACE_ID_NAMESPACE`) |
| 브리핑 내용이 예전과 동일 | 동기화 후 캐시/미반영 | `cms:sync` 재실행 → Supabase `app_config`/`places` 직접 확인 |

### 4.5 재동기화 절차 (Full Resync)

데이터 불일치·장애 복구 시:

1. Sheets 원본 데이터 정합성 확인 (archived 행, JSON 문법)
2. Dry-run으로 스킵 행 0건 확인
   ```bash
   pnpm run cms:sync:dry-run
   pnpm run cms:sync:config-dry-run
   ```
3. Full sync 실행
   ```bash
   $env:SYNC_EXECUTE="true"; pnpm run cms:sync
   ```
4. Supabase 대시보드에서 `places`, `app_config` row count·최신 `updated_at` 확인
5. `pnpm run test:webhook`으로 브리핑 URL 생성 검증
6. 프로덕션 배포 환경이라면 `APP_BASE_URL`이 올바른지 확인

### 4.6 Fixture-only 긴급 운영

Google Sheets 접근 불가 시:

```bash
pnpm run db:seed    # places만 fixture로 복구
```

config는 `DEFAULT_APP_CONFIG`로 fail-over. Sheets 복구 후 정상 sync 절차로 복귀.

|------|------|
| 피드백 누적 활용 | P1 목표 — `feedback_events` → `past_feedback` 기반 추천 반영 검증 |
| Receptivity 프로토타입 | P2 convexity — generic event schema 기반 Mind PT 모델 프로토타입 |
| 데이터 자산 | `feedback_events`의 `subject_id`, `trip_id`, `context_tags`, `satisfaction`, `failure_reason` 스키마는 이미 이식 가능하게 설계됨 |

**현재 구현 상태:** `feedback_events` insert API·동적 피드백 URL 완료. Mind PT 모델 코드는 **미착수**.

### 5.3 CRM 이식 준비 (중장기)

**설계 의도:** `feedback_events`는 송밀당 O2O CRM(L0)에 그대로 이식 가능한 generic event schema.

| 준비 완료 | 미착수 |
|-----------|--------|
| `feedback_events` DDL + insert API | CRM 커넥터/어댑터 |
| `context_tags` JSONB 스키마 (PRD §3) | subject_id ↔ CRM 고객 ID 매핑 |
| Vitest 스키마 검증 테스트 | 이벤트 스트리밍/웹훅 연동 |

### 5.4 SEED 자동화 (장기)

PRD §6 — YouTube CURTAIL + Gemini 태깅 + Google Geocoding → places import 파이프라인.  
현재 v0는 **수동 fixture/Sheets**로 Safe Pool 운영 중.
